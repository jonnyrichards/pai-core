#Create-roadmap skill Update Challenge

##I have a challenging coding task for you. As you can find in our skills dir (roadmap-create, roadmap-render), we've been working on two complementary skills to help create roadmaps. They're working reasonably well: the basic idea is the create skill generates a .md which captures the data for the roadmap, and the renderer uses a cli tool to generate the html. The problem is: I wasn't happy with the design of the html. Since creating the skills, I've had a session with Claude Design trying to design a better template. I'm now happy with the template I've created using Claude Design. The goal is to adapt our renderer so it renders a template exactly like the one Claude Design has created (or as close as possible).

##There will be several steps involved with this.
My sense of the basic approach (and you should sanity check this) is:
- Read and understand the Claude Design template - in particular all the data that is required to generate it (we won't be adapting it; it's a guide for how we change our own template and skills)
- Update our .md file (the data source for our existing roadmap template) so that it contains everything it needs to render a template like the one in Claude Code
- Update the renderer so the outputted .html reflects the template (you may want to start from scratch here rather than update what we have)
- Update the create-roadmap skill (which guides a back and forward with the user) so that at the end of that back and forward we generate a .md like the one we've just created - which can power the Claude Design template
Does that make sense?

##Some things I like about the Claude Design template:
- design overall is :chef_kiss
- swim lanes have colours, confidence level essentially reflects more or less 'complete' versions of the colour at the level of individual cards (from deep colour, light colour, outline only, dotted outline)
- cards have fixed height
- swim lanes have variable height (allowing as many or few cards as you like to be in the lane)
- effort is reflected in card length (span) and the 'mini bar' icon
- start time (ie. when the work in the card will kick off in the quarter) is reflected with absolute positioning. In the .html this is handled with absolute positioning. (The README should explain this.)

##Some things to think through:
###Existing data in the h2-roadmap.md
- the existing data should contain most of what you need to make the roadmap in the new style. Effort (small / medium / large mapping to 2 weeks, 4 weeks, 8 weeks) should enable you to decide the width of a card (span)
- the main thing that's missing will be start date. I think the Claude Design data uses a convention like 0 (start of first quarter), 1.5 (half way through second quarter) etc. We will need to understand this, and figure out how to adapt our data. Ultimately I think our .md file, which contains a table, will need a new column which encodes the start date
###Create-roadmap skill
- the create-roadmap skill will need to be told to either ascertain or (more likely) deduce a putative sequence for the roadmap items / cards as part of the create-roadmap skill. (We could push this back on the user, or the skill could take care of this, making judgments about the optimal sequence based on the items themselves and any dependencies they may have on one another etc. Either approach could be fine)
- there are some other pieces of data in the new template: title, a sense of 'which quarter is next'(are we going into: q3, q1 etc.?). In theory we should try to update the skill to figure these out in consultation with the user

##It doesn't matter if every single property of the Claude Design template isn't included from the get go. Let's get the basic roadmap components: swimlanes, cards, and card placements - working first, and then worry about fine details like page dressing etc.
I think that's it.
